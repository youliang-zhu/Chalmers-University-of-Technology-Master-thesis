\chapter{Related Work}
\label{chap:related-work}

This chapter positions NanoMem within prior work on long-term memory for LLM
agents, query-time retrieval, temporal reasoning, evidence synthesis, and
reinforcement learning for retrieval policies. The purpose is not only to list
systems that are adjacent to NanoMem, but to separate their system boundaries.
Some work asks how an agent should store or update memories before a future
query is known. Other work asks how a model should search, rerank, or reason
after the query arrives. NanoMem belongs to the latter group, but with a more
specific output contract: the memory module should return structured,
source-grounded evidence for a downstream answer model, not merely raw sessions
or a final answer.

\section{From Memory Systems to Evidence Synthesis}
\label{sec:rw-map}

Research on agent memory now spans several connected problems. Long-term memory
systems study how user histories should be persisted, compressed, indexed, and
updated across sessions. Retrieval systems study how the current query should
select useful parts of that history. Temporal memory systems study how event
order, relative dates, stale facts, and updated plans should be represented.
Reflective RAG and search agents study how a model can retrieve repeatedly
instead of relying on one fixed context. Faithful evidence synthesis studies how
retrieved text can be compressed without losing attribution or inventing facts.
Finally, reinforcement learning has been used to train memory managers,
retrievers, search agents, and answer models when supervised next-token
prediction is not enough.

These directions overlap, but they are not interchangeable. A system that
organizes memory well at ingestion time may still return the wrong evidence for
a query whose useful retrieval cue is hidden behind a temporal or causal
dependency. A system that performs multi-step search may still expose only raw
snippets or answer-oriented reasoning, leaving the downstream model to decide
which facts are grounded. NanoMem is motivated by this gap. It treats query-time
memory access as evidence-state construction: each retrieval round should add
compact evidence records that preserve source identifiers, session time, event
time, and a sufficiency verdict.

\section{Long-Term Agent Memory Systems}
\label{sec:rw-long-term-memory}

Long-term agent memory begins with the observation that an LLM's active context
cannot hold all information needed across a long-running interaction. A user
history may contain preferences, corrections, project decisions, future plans,
outdated facts, and events mentioned only once. Memory systems therefore add an
external substrate that stores information beyond the current prompt and makes
it available to future agent actions.

One line of work frames memory as context management. MemGPT treats memory as a
virtual context problem, where an agent decides which information should remain
in the active context and which should be paged to external storage
\cite{packer2023memgpt}. This view is important because it makes the context
window a managed resource rather than a fixed limitation. However, it does not
focus on constructing temporally grounded evidence records at query time.

Production-oriented memory systems instead expose persistent memory as a
service. Mem0 and MemOS organize histories into reusable memory units and
provide infrastructure for storing, updating, and retrieving user information
\cite{chhikara2025mem0,li2025memos}. These systems address practical deployment
questions such as deduplication, updates, and reuse across agent tasks. Their
typical output is a memory record or a set of retrieved memories, whereas
NanoMem's output is a structured evidence state with explicit temporal fields
and source identifiers.

Other work enriches the organization of memory itself. A-Mem constructs dynamic
note-style memories for agents; GMemory and MemSim study memory organization and
simulation; Zep uses temporal knowledge graphs; HyperMem explores hypergraph
memory for long conversations; and MemPalace represents a local-first memory
management direction \cite{xu2025amem,zhang2025gmemory,zhang2025memsim,
rasmussen2025zep,yue2026hypermem,mempalace2026}. These systems show that memory
is not only a vector index: linking, graph structure, time, and evolving
records matter. The remaining tension is that write-time extraction must decide
what to store before the future query is known. If a future query depends on a
hidden temporal bridge, the relevant interpretation may be easier to construct
after the query arrives.

Compression-oriented memory work is complementary. LightMem and R3MEM study how
long histories can be represented more efficiently or reversibly
\cite{fang2025lightmem,wang2025r3mem}. This reduces storage and context cost,
but compression alone does not solve evidence addressability. A compressed
record can still omit the intermediate cue needed for a later retrieval round,
or merge observed facts with inferred ones in a way that makes grounding hard to
inspect.

\section{Search-Side Memory Retrieval and Reflective Retrieval}
\label{sec:rw-search-side}

Write-side organization is only one part of memory. Once a query arrives, the
system must decide what to retrieve, whether the retrieved material is enough,
and what form should be passed downstream. Passive retrieval is the simplest
case: an embedding retriever or keyword system returns the top-ranked chunks or
sessions for the original query. This is scalable and general, but it assumes
that the query text already contains the right retrieval address. Temporal and
causal memory questions often violate this assumption.

A second family learns ranking or selection proxies. Temporal memory systems
such as Memory-T1 train agents to select temporally consistent evidence
sessions for multi-session reasoning \cite{du2026memoryt1}. Such methods improve
the candidate set and can reduce the burden on the answer model, but the output
is still often session-level or answer-oriented. NanoMem instead emphasizes
event-level evidence synthesis: retrieved sessions are transformed into
structured evidence items that can become search state in later rounds.

Iterative and reflective retrieval systems are closer to NanoMem's search
behavior. Self-RAG teaches a model to reflect on retrieval and generation
decisions; Search-R1 trains multi-step search behavior with reinforcement
learning; SmartRAG, RAG-RL, RPO, SSFO, TreeGRPO, and Stratified GRPO each study
ways to adapt retrieval, preference optimization, or search-agent training
\cite{asai2024selfrag,jin2025searchr1,gao2025smartrag,huang2025ragrl,
yan2025rpo,tang2025ssfo,ji2026treegrpo,zhu2025stratifiedgrpo}. In long-context
and memory settings, SUMER shows that goal-directed search can outperform
goal-agnostic compression, while MemR$^3$ introduces a reflective
retrieve--reflect--answer controller for long-term conversational memory
\cite{zheng2025sumer,du2025memr3}.

The shared insight is that retrieval should be adaptive. The difference is the
loop state and output boundary. Reflective RAG systems often carry snippets,
reasoning traces, or textual gap descriptions, and many directly produce the
final answer. NanoMem carries a Temporal Evidence Pool. Each event records what
was found, where it came from, when the source session occurred, when the event
itself occurred, and whether the current evidence is sufficient. This separates
memory evidence construction from downstream answer generation.

\section{Temporal Reasoning in Dialogue and Agent Memory}
\label{sec:rw-temporal}

Temporal reasoning is especially difficult in conversational memory because
time appears in several forms at once. A session has a timestamp, but the events
mentioned in that session may have happened earlier, may be planned for the
future, or may be described only through relative expressions such as "last
week" or "after the meeting". Long-term dialogue also contains corrections and
updates, so the latest mention is not always the same as the event that matters.
Benchmarks such as TimeBench, Test of Time, LoCoMo, and LongMemEval show that
LLMs and memory systems remain brittle when they must handle event order,
temporal constraints, and long multi-session histories
\cite{chu2024timebench,fatemi2025testoftime,maharana2024locomo,wu2024longmemeval}.

Existing systems use time in different ways. Zep encodes temporal information in
a temporal knowledge graph, making time part of the memory substrate
\cite{rasmussen2025zep}. TReMu builds timeline summaries and combines them with
neuro-symbolic reasoning for multi-session temporal questions
\cite{ge2025tremu}. Memory-T1 uses temporal-aware training signals to improve
multi-session temporal reasoning and evidence selection \cite{du2026memoryt1}.
These approaches demonstrate that time cannot be reduced to semantic
similarity. They also illustrate three design choices: time can be stored as
metadata, summarized into a timeline, or used as a retrieval and selection
signal.

NanoMem uses time as a query-time search state. The difference matters when
session time and event time diverge. If a user says on April 2 that they
travelled to Europe last week, the source session belongs to April 2, while the
travel event belongs to the previous week. A future question may need either
date depending on its phrasing. NanoMem therefore keeps both the source
session-time and the inferred event-time in the evidence record. Once inferred,
the event-time can become a retrieval cue for another round, for example when a
question first requires identifying the week of a promotion and then searching
for a book mentioned during that week.

\section{Evidence Synthesis, Faithfulness, and Source Grounding}
\label{sec:rw-evidence-synthesis}

Retrieval is not the same as evidence. Raw sessions are often long, noisy, and
contain multiple unrelated events. Passing them directly to a downstream answer
model makes the answer model perform denoising, temporal interpretation,
evidence selection, and final reasoning at the same time. This can work when the
full history fits in context and the relevant evidence is obvious, but it makes
errors hard to diagnose. If the final answer is wrong, it is unclear whether the
retriever missed the right session, the compression step hallucinated a fact, or
the answer model ignored the evidence.

Evidence synthesis addresses this boundary by converting retrieved text into
compact, inspectable records. For NanoMem, a useful evidence record should state
the supported event, identify the source session, preserve the session time,
separate the event time, and make clear whether the evidence is sufficient for
the downstream answer model. This differs from ordinary summarization because
the goal is not a fluent standalone answer. The goal is a memory-side artifact
that remains attributable and useful for subsequent retrieval.

Faithfulness is especially important when the system compresses cross-session
reasoning. A model may infer that two events are connected, but if that
inference is written as if it were directly observed, downstream users and
evaluators cannot tell what is supported by source text. NanoMem's structured
events and sufficiency verdict are designed to make this boundary explicit. A
partial event can be retained even when the verdict is insufficient, allowing
the Planner to use the partial evidence as a bridge instead of forcing an
answer prematurely.

\section{Reinforcement Learning for Memory and Retrieval Policies}
\label{sec:rw-rl}

Reinforcement learning has become relevant to memory and retrieval because many
important decisions are not well captured by next-token likelihood. A memory
system may need to learn when to write, update, retrieve, continue searching,
stop, or compress. The reward can be attached to the final answer, to retrieved
evidence, or to intermediate process decisions.

Write-side RL and policy-learning work trains memory operations before or while
memories are constructed. Memory-R1 trains agents to manage and use memories
through reinforcement learning, while MemReader and Mem-$\alpha$ study active or
learned memory construction \cite{yan2025memoryr1,kang2026memreader,
wang2025memalpha}. MMOA-RAG and EMG-RAG also connect retrieval-augmented
generation with multi-objective or external-memory optimization
\cite{chen2025mmoarag,wang2024emgrag}. These works show that memory operations
can be optimized as policies, but their action spaces differ from NanoMem's
query-time evidence construction.

Search-side RL optimizes retrieval and reasoning after a query arrives.
Search-R1, RAG-RL, RPO, SSFO, TreeGRPO, and Stratified GRPO train or optimize
multi-step retrieval and search agents \cite{jin2025searchr1,huang2025ragrl,
yan2025rpo,tang2025ssfo,ji2026treegrpo,zhu2025stratifiedgrpo}. These methods
are relevant because NanoMem also depends on multi-step search. However, many
search-agent rewards are dominated by final answer correctness or generic
search success. For temporal-causal memory, an intermediate retrieval round can
be useful even when it cannot yet answer the question: it may identify the
temporal bridge needed for the next round.

Memory-T1 is particularly close because it studies temporal reasoning in
multi-session memory with reinforcement learning \cite{du2026memoryt1}. The key
distinction is the trained object and output. Memory-T1 primarily improves
session selection and answer behavior for temporal questions, while NanoMem
trains the Synthesizer as an intermediate evidence construction policy. Its
reward design supervises parseable format, compactness, downstream answer
usefulness, and the verdict that decides whether to stop or continue retrieval.

\section{Benchmark Coverage}
\label{sec:rw-benchmarks}

The evaluation of long-term memory systems requires more than one benchmark
because different datasets stress different capabilities. LoCoMo provides
multi-session long histories and includes single-hop, multi-hop, temporal, and
open-domain questions \cite{maharana2024locomo}. LongMemEval focuses on
chat-assistant memory and includes practical cases such as single-session
recall, multi-session reasoning, temporal reasoning, knowledge updates, and
abstention \cite{wu2024longmemeval}. These benchmarks are necessary because
they test whether a memory system remains useful in broad long-term
conversational settings.

Temporal reasoning benchmarks such as TimeBench and Test of Time isolate
temporal robustness more directly \cite{chu2024timebench,fatemi2025testoftime}.
They help explain why temporal normalization and event ordering are difficult,
but they do not by themselves capture the full agent-memory setting with
session sources, retrieved evidence, and query-time evidence construction.

NanoMem also uses ChainMem as a diagnostic benchmark for indirectly addressable
evidence. Its purpose is to test cases where the next useful retrieval cue must
be inferred from previous evidence, such as finding a temporal bridge before
retrieving the answer-bearing session. ChainMem therefore complements LoCoMo and
LongMemEval rather than replacing them: public long-memory benchmarks test
generality, while ChainMem stresses the evidence addressability gap.

\section{Research Gap}
\label{sec:rw-gap}

The reviewed work shows that NanoMem builds on several mature directions rather
than replacing them. Write-side memory systems help store, compress, link, and
update long histories, but they must make many decisions before the future query
is known. Search-side and reflective retrieval systems show that adaptive
retrieval is useful, but their intermediate state is often raw text, reasoning
traces, or answer-oriented context. Temporal memory systems make time explicit,
but often treat it as metadata, a timeline, or a selection signal rather than as
evidence state that can drive the next retrieval round. RL-based memory and RAG
work demonstrates that memory and retrieval policies can be trained, but
terminal answer rewards alone give weak supervision for whether the current
evidence is sufficient.

NanoMem therefore occupies a narrower research gap: query-time evidence
synthesis for temporal-causal agent memory. It asks how a memory module can
iteratively construct a Temporal Evidence Pool, preserve event time and session
time separately, expose the source of each evidence item, and use a verdict to
decide whether to continue searching. The following chapter develops this
method in detail.
